📊 Package store

internal/store/

The database access layer. Each store struct owns all queries for one domain entity, communicates with PostgreSQL via the pgx/v5 driver, and returns typed Go structs. All methods accept context.Context for cancellation and timeout propagation.

Design Principles

  • All queries use parameterized arguments — no string interpolation in SQL.
  • Store methods return domain model structs, not raw database rows.
  • Stores are injected into handlers as interfaces, allowing mock-free testing with a real test database.
  • Transactions are used where multiple statements must succeed or fail together (e.g. LobbyStore.Create).

UserStore

Central store for user accounts. Handles the full lifecycle from registration through profile updates.

User Model

type User struct {
    ID                         string
    Email                      string
    PasswordHash               string
    DisplayName                string
    AvatarURL                  string
    EmailVerified              bool
    VerificationToken          *string    // SHA-256 hash, not raw token
    VerificationTokenExpiresAt *time.Time
    PasswordResetToken         *string    // SHA-256 hash
    PasswordResetExpiresAt     *time.Time
    Status                     string    // online|offline|away|in_game|in_lobby|custom
    CustomStatus               string
    NotificationsEnabled       bool
    CreatedAt, UpdatedAt       time.Time
}

Methods

  • Create(ctx, user)Inserts a new user row with a verification token hash.
  • GetByID(ctx, id)Looks up a user by UUID. Returns pgx.ErrNoRows if not found.
  • GetByEmail(ctx, email)Case-insensitive lookup via functional index on lower(email).
  • GetByVerificationToken(ctx, hash)Looks up an unverified user by their token hash. Used during email verification.
  • GetByPasswordResetToken(ctx, hash)Looks up a user by password-reset token hash. Checks expiry.
  • VerifyEmail(ctx, userID)Sets email_verified = true, clears the verification token fields.
  • SetPasswordReset(ctx, userID, hash, expiry)Stores a new password-reset token hash with a 1-hour expiry.
  • ResetPassword(ctx, userID, newHash)Updates the password hash and clears the reset token fields.
  • UpdateProfile(ctx, userID, displayName, avatarURL)Updates display name and/or avatar URL. Either field can be empty to skip updating it.
  • UpdateStatus(ctx, userID, status, customStatus)Updates the user's presence status.
  • IsEmailTaken(ctx, email)Returns true if a non-deleted user exists with this email.
  • IsDisplayNameTaken(ctx, name)Returns true if a non-deleted user exists with this display name.
  • SearchByDisplayName(ctx, query, limit)Trigram similarity search using the pg_trgm extension's GIN index.

FriendStore

Manages the directed friendship graph. A friendship record has a direction (from → to) and a status (pending/accepted/blocked). Accepted friendships are symmetric — both users appear in each other's friend lists.

Methods

  • ListFriends(ctx, userID)Returns all accepted friends with their profile (displayName, avatarURL, status).
  • ListPendingRequests(ctx, userID)Incoming requests where to_user_id = userID and status is pending.
  • ListSentRequests(ctx, userID)Outgoing requests where from_user_id = userID and status is pending.
  • SendRequest(ctx, fromID, toID)Inserts a pending friendship. Errors if blocked or already friends.
  • RespondToRequest(ctx, friendshipID, userID, accept bool)Changes status to accepted or declined. Validates userID is the recipient.
  • RemoveFriend(ctx, friendshipID, userID)Deletes the friendship row. Both users lose each other from their lists.
  • BlockUser(ctx, fromID, toID)Upserts a blocked status. Removes any existing friendship.
  • GetFriendIDs(ctx, userID)Returns a slice of UUIDs of accepted friends. Used by the presence hub to target broadcasts.
  • IsFriend(ctx, userID, otherID)Returns true if an accepted friendship exists between the two users.
  • IsBlocked(ctx, userID, otherID)Returns true if either user has blocked the other.

GameStore

Read-only access to the game catalogue seeded by migration 000011.

  • ListActive(ctx)All games where is_active = true, ordered by title.
  • GetByID(ctx, id)Single game by UUID.
  • GetBySlug(ctx, slug)Single game by URL slug.

LobbyStore

Manages the full lobby lifecycle from creation through game start. Create uses a transaction to insert the lobby and add the host as the first member atomically.

  • Create(ctx, lobby, hostID)Transaction: inserts lobby + inserts host as member with is_ready=false.
  • GetByID(ctx, id)Single lobby row.
  • GetWithMembers(ctx, id)Lobby row joined with member rows and user profiles.
  • List(ctx)All public lobbies with status "waiting", with game info joined.
  • Delete(ctx, id, hostID)Sets closed_at. Validates caller is host.
  • Join(ctx, lobbyID, userID)Inserts a member row. Fails if lobby is full or not waiting.
  • Leave(ctx, lobbyID, userID)Removes a member row.
  • UpdateReady(ctx, lobbyID, userID, isReady)Sets the member's is_ready flag.
  • UpdateStatus(ctx, id, status)Changes lobby status (waiting → in_game → finished).
  • AllReady(ctx, id)Returns true if all members have is_ready = true.

MessageStore

  • Create(ctx, msg)Inserts a message and immediately joins with the sender's profile (displayName, avatarURL) in one query for efficient return.
  • ListByChannelID(ctx, channelTypeID, channelID)All messages in a channel (e.g. a specific lobby ID), ordered by sent_at ASC.

PushSubscriptionStore

  • Upsert(ctx, sub)Insert or update by endpoint. Transfers ownership if the endpoint belonged to another user.
  • DeleteByEndpoint(ctx, userID, endpoint)Removes the subscription for this user+endpoint pair.
  • ListByUserID(ctx, userID)All active subscriptions for a user (they may have multiple devices/browsers).

RefreshTokenStore

Implements server-side refresh token rotation with family-based revocation. A "family" is a chain of rotated tokens — if any token in the family is reused after being replaced, the entire family is revoked (detecting token theft).

  • Create(ctx, token)Inserts a new refresh token row with a SHA-256 hash, family ID, and 7-day expiry.
  • ValidateAndRotate(ctx, hash)Finds the token by hash, validates it's not expired/revoked, marks it revoked, returns the user ID and family ID so a new token can be issued for the same family.
  • Revoke(ctx, familyID)Marks every token in a family as revoked (token reuse detection or explicit logout).
  • DeleteExpired(ctx)Deletes rows older than their expires_at — run periodically by a cron job or startup task.
