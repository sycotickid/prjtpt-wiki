❤️ Package health

internal/health/

Provides the GET /health endpoint used by Docker Compose health checks, load balancers, and orchestrators to determine if the application instance is ready to serve traffic.

Files

FileResponsibility
health.goHandler that pings the database and returns a JSON health status response.
health_test.goUnit tests for the health handler using a mock database pinger.

Endpoint

GET /health Returns 200 OK with JSON body if the server and database are reachable. Returns 503 Service Unavailable if the database ping fails.

Response Body (healthy)

{
  "status": "ok",
  "database": "ok"
}

Response Body (unhealthy)

{
  "status": "degraded",
  "database": "unreachable"
}

How It's Used

The health endpoint is consumed by multiple layers of the stack:

  • Docker Compose: The api service defines test: wget -q -O- http://localhost:8080/health as its healthcheck. Other services that depend on the API wait for this to pass.
  • Caddy: Excludes /health from access logging to avoid cluttering logs with probe noise.
  • Orchestrators: Kubernetes or ECS can use this endpoint as a readiness probe. A failing database will surface the instance as not-ready and stop routing traffic to it.

Readiness vs liveness: This endpoint acts as both. It's "live" if the HTTP server responds, and "ready" only if the database ping succeeds. In a Kubernetes deployment you might split these — the liveness probe would hit any route, and the readiness probe would hit /health.
