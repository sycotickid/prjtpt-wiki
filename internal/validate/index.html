internal/validate/

Lightweight field-level input validation. Each function validates a single field and returns either an empty string (valid) or a human-readable error message. An Errors map type collects multiple field errors for API responses.

Files

FileResponsibility
validate.goField validator functions and the Errors type for collecting per-field validation messages.

Validators

  • Email(v string) string Required. Maximum 256 characters. Must match something@something.something pattern. Returns error message or empty string.
  • Password(v string) string Required. 8–72 characters (bcrypt silently truncates above 72). Must contain at least one letter and at least one digit.
  • DisplayName(v string) string Required. 3–32 characters. Only alphanumeric characters, underscores, and hyphens allowed. No spaces.

Errors Type

type Errors map[string]string

// Usage in a handler:
errs := validate.Errors{}

if msg := validate.Email(req.Email); msg != "" {
    errs["email"] = msg
}
if msg := validate.Password(req.Password); msg != "" {
    errs["password"] = msg
}
if len(errs) > 0 {
    respond.jsonValidationError(w, errs) // 422 Unprocessable Entity
    return
}

API Response for Validation Failure

{
  "error": "validation failed",
  "fields": {
    "email": "must be a valid email address",
    "password": "must be at least 8 characters"
  }
}

Design Philosophy

The validate package is intentionally minimal — no reflection, no struct tags, no external dependencies. Validators are pure functions: given a string, return an error string or nothing. This makes them trivial to test, compose, and extend.

Validation boundaries: Validation is applied at the HTTP handler boundary only (on user input). Internal function calls between packages do not re-validate — trust is established at the edge. This avoids redundant error handling deep in the call stack.
