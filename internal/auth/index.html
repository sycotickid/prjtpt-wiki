🔐 Package auth

internal/auth/

Provides all cryptographic primitives used for authentication: RS256 JWT issuance and validation, bcrypt password hashing, and secure random token generation for email verification and password-reset flows.

Files

jwt.go — JWTService

Wraps golang-jwt/jwt/v5 with RS256 (asymmetric RSA) signing. The private key signs tokens; the public key validates them. This means validation can be distributed to services that never hold the private key.

Constructor

func NewJWTService(privateKeyPEM, publicKeyPEM string) (*JWTService, error)

Parses PEM-encoded RSA keys from environment variables. Supports literal \n sequences (common in .env files) so keys can be stored as single-line strings.

Methods

  • IssueAccessToken(userID string) (string, error) Creates a signed JWT with the user's ID as the sub claim and a 15-minute expiry. Returns the compact serialised token string.
  • ValidateAccessToken(token string) (string, error) Parses and validates the token's signature, expiry, and issuer. Returns the sub (user UUID) on success, or an error if expired/tampered.

JWT Claims Structure

{
  "sub": "<user-uuid>",
  "iss": "prjtpt",
  "iat": 1712000000,
  "exp": 1712000900   // +15 minutes
}

Why RS256? Asymmetric signing means the API server's public key can be shared with any service that needs to verify tokens without exposing the private key used to issue them. HS256 (shared secret) would require every service to hold the same secret.

password.go — Password Hashing

Thin wrappers around golang.org/x/crypto/bcrypt. bcrypt's built-in salt and iteration count make rainbow-table and brute-force attacks computationally expensive.

Functions

  • HashPassword(plain string) (string, error) Returns a bcrypt hash at cost 12. Higher cost = slower hashing = harder brute-force. bcrypt automatically generates a random salt and embeds it in the hash string.
  • CheckPassword(plain, hash string) error Constant-time comparison of a plaintext password against its bcrypt hash. Returns nil on match, bcrypt.ErrMismatchedHashAndPassword on failure. Constant-time prevents timing attacks.

bcrypt input limit: bcrypt silently truncates inputs longer than 72 bytes. The validate package enforces a 72-character maximum on passwords at the API boundary to ensure what the user typed is exactly what gets hashed.

token.go — Secure Token Generation

Used for email-verification tokens and password-reset tokens. The token sent to the user and the token stored in the database are different values — the stored value is a SHA-256 hash of the raw token, so a database breach cannot be used to exploit the tokens.

Functions

  • GenerateToken() (raw, hash string, err error) Reads 32 bytes from crypto/rand, base64url-encodes them as raw, and returns hash = SHA-256(raw). Send raw to the user; store hash in the DB.
  • HashToken(raw string) string Deterministically hashes a raw token string. Used when a user presents a token — hash it first, then query the DB for the hash.

Token Flow

// On email verification request:
raw, hash, _ := auth.GenerateToken()
userStore.StoreVerificationTokenHash(ctx, userID, hash, expiry) // store hash
mailer.SendVerification(email, appURL+"/verify?token="+raw)      // send raw

// When user clicks the link:
hash := auth.HashToken(r.URL.Query().Get("token"))
user, err := userStore.GetByVerificationToken(ctx, hash)           // look up hash
