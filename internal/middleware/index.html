internal/middleware/

HTTP middleware providing cross-cutting concerns: CORS enforcement, structured request logging, OWASP security headers, JWT authentication, and double-submit CSRF protection.

cors.go — CORS Enforcement

Implements single-origin CORS rather than a wildcard allow-list. Only the origin in cfg.CORSOrigin is permitted.

Behaviour

  • OPTIONS preflight: Responds immediately with 204 No Content and the appropriate CORS headers, without forwarding to the handler.
  • Allowed origin: Adds Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Headers, and Access-Control-Allow-Credentials: true.
  • Disallowed origin: Returns 403 Forbidden immediately.
  • No Origin header: Allowed through (server-to-server or same-origin requests).

Credentials flag: Access-Control-Allow-Credentials: true is set because the application uses HttpOnly cookies for refresh tokens. This is safe because the origin is restricted to a single known value, not a wildcard.

auth.go — JWT Middleware

Extracts a Bearer token from the Authorization header, validates it via JWTService, and stores the authenticated user ID in the request context. Protected routes use RequireAuth as a middleware in their route group.

Exported Symbols

  • RequireAuth(jwtSvc *auth.JWTService) func(http.Handler) http.Handler Middleware factory. Returns 401 Unauthorized if no token is present or it fails validation. Stores the user UUID in context on success.
  • UserIDFromContext(ctx context.Context) (string, bool) Retrieves the authenticated user ID set by RequireAuth. Handlers call this rather than repeating the context key lookup.
  • WithUserID(ctx context.Context, userID string) context.Context Test helper — injects a user ID into a context so handlers can be tested without a real JWT.

csrf.go — Double-Submit Cookie

Implements the double-submit cookie CSRF mitigation pattern. A random token is set as a readable (non-HttpOnly) cookie. State-mutating requests must include the same token in the X-CSRF-Token header. An attacker cannot read the cookie from a different origin, so they can't replicate the header.

Flow

  1. On the first request (no cookie present) the middleware generates a 32-byte random token (base64url-encoded), sets it as csrf_token cookie with SameSite=Strict, Secure, but not HttpOnly so JavaScript can read it.
  2. The Datastar frontend reads this cookie and includes its value as the X-CSRF-Token request header on every POST/PUT/PATCH/DELETE.
  3. The middleware compares cookie value to header value. A mismatch returns 403 Forbidden.
  4. Safe methods (GET, HEAD, OPTIONS) are not checked.
